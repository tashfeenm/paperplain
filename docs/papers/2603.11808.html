Automating Skill Acquisition through Large-Scale Mining of Open-Source Agentic Repositories: A Framework for Multi-Agent Procedural Knowledge Extraction

Shuzhen Bi, Mengsong Wu, Hao Hao, Keqian Li, Wentao Liu, Siyu Song, Hongbo Zhao, Aimin Zhou

View original paper on arxiv ↗
TL;DR The Idea Why It Matters What's Missing

TL;DR

AI assistants like ChatGPT are good generalists, but they struggle with specialized tasks that require step-by-step expertise. This paper proposes a system that automatically mines open-source code repositories on GitHub to extract reusable "skills" -- packaged bundles of know-how that any AI agent can pick up and use without retraining the entire model. Think of it like building a library of recipe cards that any chef can follow, instead of trying to teach every chef every dish from scratch. The researchers demonstrate this by extracting educational video-making skills from two existing projects, whose own benchmarks show 40% better knowledge transfer than baseline approaches.

The Idea

Today's large language models know a lot of facts, but they often fumble when asked to execute a complex, multi-step procedure -- like generating an animated math tutorial or debugging a visualization pipeline. The traditional fix is to retrain or fine-tune the model, which is expensive and slow. This paper proposes a different approach: instead of baking expertise into the model's weights, package it as a portable "skill file" that any AI agent can load on demand. It is the difference between surgically implanting a memory versus handing someone an instruction manual.

The framework works in three stages. First, a tool maps out the structure of a GitHub repository -- its scripts, config files, and documentation -- to understand how the codebase is organized. Second, a retrieval system uses embedding-based search (similar to how search engines match queries to documents) to identify reusable procedural patterns buried in the code. It looks for logic that is recurring, non-obvious, and generalizable -- not just one-off hacks. Third, these patterns are translated into a standardized format called SKILL.md, which has three layers: lightweight metadata (so agents know the skill exists), detailed instructions (loaded only when needed), and bundled scripts and templates (loaded on demand). This layered design is clever because an agent can be aware of thousands of skills without clogging its limited working memory.

The researchers demonstrate this pipeline on two real projects: TheoremExplainAgent, which generates animated video explanations of math and science theorems, and Code2Video, which turns code into educational videos. From these, they extract skills like "Visual Theorem Walkthrough" (generate step-by-step animated proofs) and "Visual Layout Critic" (automatically check whether a rendered frame has overlapping text or cluttered elements). Both skills are packaged as portable, reusable units that any compatible AI agent can pick up.

A notable contribution is the security governance framework. Since skills are mined from public repositories, they could contain malicious code. The paper proposes a four-stage verification pipeline: static analysis (scanning for dangerous patterns like eval()), semantic classification (using an LLM to check for hidden prompt injections), sandboxed execution (running scripts in an isolated container), and permission validation (ensuring the skill only accesses what it claims to need). Skills graduate through trust tiers based on verified safe operation -- similar to how app stores review submissions before publishing.

Why It Matters

This work addresses a real bottleneck in making AI agents practically useful. Right now, giving an AI assistant a new specialized capability typically means either retraining the model (expensive, slow, requires expertise) or writing custom integrations (labor-intensive, fragile). If you could instead point a system at GitHub and have it automatically extract, package, and verify reusable skills, you dramatically lower the cost of making agents competent in new domains. Domain experts who build great open-source tools -- but never intended them for AI consumption -- suddenly become indirect contributors to the AI skill ecosystem.

The educational content angle is particularly compelling. The paper shows that agent-generated tutorial videos, built using extracted skills, achieve 40% better knowledge transfer than baseline approaches. If these results hold broadly, it could accelerate the creation of high-quality STEM education materials -- especially for topics where human-crafted tutorials are scarce or outdated. More broadly, the SKILL.md standard and the extraction pipeline together sketch a future where AI capabilities grow through a package-manager-like ecosystem: discover, install, verify, compose. That is a meaningfully different scaling story than "train a bigger model."

What's Missing

The paper's demonstration is narrow. Both source repositories (TheoremExplainAgent and Code2Video) are in the same domain -- Manim-based educational animation -- and were likely chosen because their code is well-structured and their outputs are easy to evaluate. The framework's effectiveness on messier, less-documented repositories (which is most of GitHub) remains unproven. The 40% knowledge transfer improvement comes from the Code2Video paper's own benchmark (TeachQuiz), not from an independent evaluation of the extraction pipeline itself, so it is hard to attribute that gain specifically to the skill extraction process.

The security governance framework, while sensible in design, is described at an architectural level without empirical validation. The paper cites a statistic that 26.1% of community-distributed skills contain vulnerabilities, but does not report how well the proposed four-stage pipeline catches them. The dense retrieval approach for identifying latent skills also lacks quantitative evaluation -- there are no precision/recall numbers showing how accurately the system distinguishes genuinely reusable patterns from project-specific code. Finally, the paper does not address skill maintenance: open-source libraries evolve, APIs break, and dependencies shift. How extracted skills stay current as their source repositories change is an open question the paper acknowledges only in passing.
