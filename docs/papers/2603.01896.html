AI Agents March 2026

Agentic Code Reasoning

Shubham Ugare, Satish Chandra

View original paper on arxiv ↗
TL;DR The Idea Why It Matters What's Missing

TL;DR

When AI agents try to understand code, they often take shortcuts -- guessing what a function does based on its name instead of actually tracing through the logic. Researchers at Meta found that giving AI agents a structured "fill-in-the-blank" template (they call it semi-formal reasoning) forces them to show their work step by step, much like a math teacher requiring students to write out proofs. This simple change boosted accuracy across three different code analysis tasks by 5 to 12 percentage points, all without ever running the code. The best result: 93% accuracy at judging whether two code patches do the same thing (using the strongest model, Opus-4.5).

The Idea

Imagine you're a detective investigating whether two different fixes to the same software bug actually behave the same way. You could glance at both fixes and say "yeah, they look similar enough." Or you could methodically trace through every line, check what each function actually does, and build a case. Most AI coding agents today do the equivalent of glancing -- they use free-form "chain of thought" reasoning that lets them skip steps and make assumptions. This paper introduces a better approach.

The researchers created structured templates -- essentially worksheets -- that force AI agents to fill in specific sections before reaching a conclusion. For checking whether two patches are equivalent, the template requires: state what each patch does (premises), trace how each test would behave with each patch applied (claims), identify any edge cases the tests exercise, and provide either a concrete counterexample or proof that none exists before writing a final answer. It's the difference between an essay exam and a structured lab report.

The key insight comes alive in a concrete example from Django. Two patches both try to fix date formatting for years before 1000 CE. One uses format(self.data.year, "04d"), the other uses '%02d' % (self.data.year % 100). A standard AI agent assumes format() is Python's built-in and declares them equivalent. But the semi-formal template forces the agent to actually look up the function definition -- and discovers that Django has its own module-level format() function that expects a datetime object, not an integer. The first patch would crash with an error. The structured template caught what free-form reasoning missed because it required the agent to verify, not assume.

The researchers tested this across three tasks: patch equivalence (do two code changes produce the same test results?), fault localization (find the buggy lines given only a failing test), and code question answering (answer nuanced questions about how code works). Semi-formal reasoning improved accuracy on all three. For patch equivalence on real-world patches, it hit 93% accuracy. For code questions, it reached 87%. For bug finding, it gained 5-12 percentage points over standard approaches.

Why It Matters

The most immediate practical application is making AI training cheaper. Today, when companies train AI coding agents using reinforcement learning, they need to actually run code in sandboxed environments to check if a proposed fix works -- spinning up containers, installing dependencies, executing test suites. That's expensive and slow. If an AI verifier can judge patch correctness at 93% accuracy without running anything, it could dramatically reduce the cost of generating training signals. The paper specifically frames this as enabling "execution-free RL reward signals."

More broadly, this work suggests a surprisingly effective middle ground between two extremes in code analysis. On one side, you have formal verification (mathematically proving code is correct), which is powerful but impractical for most real-world codebases. On the other side, you have free-form AI reasoning, which is flexible but sloppy. Semi-formal reasoning -- structured templates that force evidence-gathering without requiring full mathematical proof -- turns out to capture much of the benefit of formalism while remaining practical. For anyone building AI-powered code review tools, static analysis, or automated debugging, this is a straightforward technique that could be adopted immediately: just change the prompt to require structured evidence before conclusions.

What's Missing

The paper's biggest limitation is one the authors acknowledge: SWE-bench (a widely-used benchmark of real software engineering tasks from GitHub) data may appear in the training data of the models being tested. While they argue this affects all configurations equally (so relative comparisons are fair), it means the absolute accuracy numbers -- the headline 93% -- might not hold on truly novel code. The evaluation is also limited to Python and Java repositories from well-known open source projects, leaving open how well this generalizes to less common languages, proprietary codebases, or code that relies heavily on runtime behavior like concurrency or network calls.

There's also a subtle irony in the results. The semi-formal approach uses roughly 2-3 times as many reasoning steps as standard approaches (38 steps vs. 20 for patch equivalence). That means 2-3 times the compute cost per verification. The paper doesn't discuss this cost-accuracy tradeoff in detail. And for one model (Sonnet-4.5 on code QA), the semi-formal template provided essentially no improvement over standard agentic reasoning, suggesting the technique's value depends on the base model's existing capabilities. Finally, the error analysis reveals that even with structured templates, agents still fail on bugs involving third-party libraries (where source code isn't available to trace), multi-file bugs spanning many locations, and domain-specific algorithmic issues -- precisely the kinds of problems that matter most in real-world software engineering.
